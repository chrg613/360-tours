# Storage Strategy

This document defines how the platform stores and delivers large 360° media safely and efficiently.

Related docs:
- Canonical schemas: `../00-conventions.md`
- Backend contract: `api-specification.md`

## Goals (MVP)

- Handle high-resolution panorama uploads (15–50MB typical).
- Deliver media globally with low latency.
- Enforce access control for private tours.
- Support background processing for derivatives (thumbnails, web-optimized).

## Storage building blocks

- **Object storage**: S3-compatible (AWS S3, Cloudflare R2, MinIO), or equivalent.
- **CDN**: CloudFront, Cloudflare, Fastly, or equivalent.
- **Metadata**: stored in PostgreSQL (see `database-schema.md`).

## Object key design

The object namespace MUST support multi-tenancy and prevent accidental collisions.

Recommended prefixes:

```
tenants/{owner_id}/tours/{tour_id}/
  scenes/{scene_id}/
    original/{file_id}.jpg
    web/{file_id}.webp
    thumbnail/{file_id}.jpg
  floor-plans/{floor_plan_id}/original/{file_id}.png
  branding/logo/{file_id}.png
```

Notes:
- `file_id` is generated by the backend and is the canonical reference.
- Clients MUST treat file URLs as opaque.

## Upload flow (MVP)

The backend MUST support a presigned upload flow:

1. Client calls `POST /api/v1/uploads/presign` with filename, MIME type, and byte size.
2. Client uploads bytes to `upload_url` using the provided method/headers.
3. Client calls `POST /api/v1/uploads/complete`.
4. Backend returns a `MediaFile` record and (optionally) triggers processing.

Uploads MUST validate:
- MIME type and extension
- file size
- panorama aspect ratio (2:1 recommended)
- basic corruption checks

## Processing pipeline

Processing SHOULD be asynchronous and job-based.

Typical stages:
- Extract EXIF metadata (camera model, dimensions, GPS when present)
- Generate thumbnails
- Generate web-optimized derivatives (e.g., WebP)
- (Optional) generate higher quality/VR derivatives

## Access control and URLs

The platform supports `visibility=private|unlisted|public` (see `../00-conventions.md`).

### Private media

Private tours MUST not expose public object URLs.

Acceptable strategies:
- return short-lived signed URLs
- proxy bytes via the API (slower, but simplest)

### Public/unlisted media

Public and unlisted tours MAY return CDN URLs with long cache lifetimes.

## CDN caching

- Public/unlisted derivatives SHOULD be served with long-lived cache headers:
  - `Cache-Control: public, max-age=31536000, immutable`
- Prefer versioned object keys (e.g., `{file_id}` changes when content changes) to avoid invalidations.

## Lifecycle and retention

- Temporary multipart/staging objects MUST be cleaned up automatically.
- When a tour is deleted, objects SHOULD be retained for a short grace period (e.g., 7–30 days) before hard deletion.
- Processed derivatives MAY be regenerated and therefore MAY have shorter retention than originals.

## Security requirements

- Encrypt at rest using the storage provider’s KMS.
- Enforce TLS for all uploads/downloads.
- Consider malware scanning for user-uploaded media (Optional).
- Prevent SSRF and path traversal: storage keys are backend-generated only.

## Observability

Track at minimum:
- upload success/failure rate
- processing queue depth
- derivative generation latency
- CDN cache hit ratio
